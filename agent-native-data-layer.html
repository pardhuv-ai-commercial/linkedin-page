Executive Summary

The Quiet Revolution at the
Bottom of the Stack

For decades, enterprise databases served as silent, compliant record-keepers. The emergence of autonomous agents has ended that arrangement. Agents do not merely read data — they reason over it, write to it, invoke transactional logic, and initiate downstream workflows, often without direct human intervention. The database, long regarded as infrastructure, is becoming the substrate of enterprise cognition itself.

11% of enterprises actively using agentic AI in production
Deloitte, 2025
40% of agentic AI projects predicted to fail by 2027
Gartner forecast
S$1B in measured economic value from AI — DBS Bank 2025, with 800+ deployed models

The contrast between those three numbers captures the central tension of enterprise agentic AI: transformative value is real and already being measured — but structural failure is equally real and at least as likely. The difference, consistently, lies not in the sophistication of the models deployed but in the quality of the data infrastructure, governance, and architectural discipline underneath them.

Core Thesis

The agent-native data layer is a new architectural construct that sits between raw storage and AI orchestration. It comprises three interlocking components: core databases providing transactional, analytical, semantic, and graph capabilities; native agent layers embedded directly into database platforms by major vendors; and middle-layer platforms that broker, govern, and orchestrate agent–data interactions across heterogeneous environments. Organisations that treat these three components as a coherent, governed system will outpace those deploying isolated PoCs against unmanaged data infrastructure.

The practitioner evidence is clear: the technology is the easy part. As DBS Bank's Chief Data Officer noted, "The tech is the easiest bit. The processes, the structure, the people, the culture — that's the hard bit." Architecture choices made in the next 12 months will set the trajectory for the following five years.

Why Databases Are Becoming the Substrate for Agentic AI

Unlike batch ML pipelines or real-time scoring APIs, agents exhibit four capabilities that demand robust data infrastructure: persistence (maintaining state and memory across sessions), retrieval (dynamically fetching semantically relevant context at reasoning time), action (writing results, updating records, triggering processes), and accountability (leaving auditable trails that satisfy governance and regulatory requirements).

No LLM can satisfy these requirements alone. The context window is finite, stateless, and opaque. Consequently, the data layer is not a supporting actor — it is the environment in which agents live. A 5% error rate is acceptable for a chatbot. It is catastrophic for an agent that places orders, updates databases, or makes automated financial decisions. One corrupted database entry can halt operations. This is the practitioner's fundamental insight that separates serious architectural thinking from demonstration-grade deployment.

Five Architectural Questions Every CTO Must Answer

  1. Where do agents execute? Inside the database engine (native), in an external orchestration layer, or mediated through a middle-layer platform — each carries distinct latency, isolation, and compliance implications.
  2. How is data access governed? Which identity model propagates to the agent? How are row-level security, PII masking, and privilege escalation handled when an agent acts on behalf of a user?
  3. What is the rollback and incident response model? When an autonomous agent executes a multi-step workflow that partially fails, what are the transactional boundaries, and who or what can halt and reverse the sequence?
  4. How do we prevent hallucination from poisoning production data? Schema grounding, query validation, and output verification are architectural concerns, not prompt-engineering band-aids. LinkedIn's internal NL2SQL agent — serving 300+ weekly users with mature schema grounding — achieves only 53% accuracy on its own benchmark. This is the production reality.
  5. How do we federate across heterogeneous data systems without creating a governance blind spot? Most enterprises operate 15–40 distinct data systems. An agent that can traverse all of them is either powerful or catastrophic, depending on the controls in place.
Definition · The Agent-Native Data Layer

The ensemble of data stores, vendor-embedded agent capabilities, middle-layer platforms, and governance controls that collectively provide autonomous AI agents with persistent memory, semantic retrieval, transactional access, and auditable action — at enterprise scale and under regulatory compliance. It is not a product. It is an architectural posture.

Consistency Models and the Practitioner's Trap

A recurring failure mode in early agentic deployments is treating all data stores as interchangeable. They are not. OLTP stores offer ACID consistency but struggle with concurrent agent writes — an agent writing to an Oracle or PostgreSQL system mid-workflow can corrupt state if transaction boundaries are not explicitly managed. OLAP and lakehouse systems favour analytical breadth over transactional precision: agents should read, never ad hoc write, against production lakehouses. Vector stores operate under approximate nearest-neighbour semantics which are probabilistic — the same query may surface different results under load, a governance concern when agents use retrieved context for regulated decisions.

Section Implications — 12-Month Actions
  • Audit the consistency model of every data store that agents will access and document failure modes for partial-write scenarios.
  • Categorise stores by interaction type: read-only reasoning, memory layer, or write-capable system of action — and apply proportionate governance to each.
  • Resist designating a single "AI database" — the agent-native data layer is inherently multi-model and forcing all workloads onto one platform creates performance and governance bottlenecks.

Section 03

External Agents &
Integration Patterns

"How do we let agents act on our core systems without losing control of security, governance, and data integrity?"

MCP, A2A, and the Protocol Layer

The Model Context Protocol (MCP) defines a standard interface by which LLMs can invoke external tools — including databases — as structured services. MCP, which saw broad adoption throughout 2025, standardises how agents connect to external tools, databases, and APIs — transforming what was previously custom integration work into plug-and-play connectivity. An MCP server wraps a data source with defined schemas, input validation, and output contracts, transforming an arbitrary database into a governed tool.

The Agent-to-Agent (A2A) protocol extends this model to multi-agent systems, enabling specialised agents — query, validation, compliance — to collaborate in structured workflows. Where MCP governs tool calls, A2A governs agent coordination. At the SDK level, LangChain and LlamaIndex dominate developer mindshare; Semantic Kernel is Microsoft's enterprise-grade framework; the OpenAI Agents SDK provides lightweight abstraction.

Three Alternative Architectures

Architecture A — DB-Centric Native Agents

The LLM orchestrator is embedded within or closely coupled to a single database vendor's platform (Oracle 26ai, Snowflake Cortex, Databricks Mosaic). All agent reasoning, retrieval, and action occurs within the vendor's compute and data perimeter. Best for: organisations with deep investment in a single platform, strong security requirements, and relatively homogeneous data landscapes. Trade-off: High governance fidelity; low cross-vendor flexibility. Lock-in risk compounds as AI capabilities evolve faster than vendor release cycles.

Architecture B — Orchestrator-Centric with DB as Tool

An external orchestration framework (LangChain, Semantic Kernel, OpenAI Agents) manages reasoning and planning. Databases are invoked as tools via MCP or SDK connectors. The orchestrator owns the workflow graph; each database interaction is a bounded, governed tool call. Best for: organisations with heterogeneous data estates, multi-vendor strategies, or rapid iteration requirements. Trade-off: High flexibility; governance must be implemented at the orchestration and MCP layers, which requires engineering discipline. The most significant theme across production deployments is the systematic movement of safety logic out of prompts and into infrastructure — every time a new model emerges, prompt-level exploits emerge within hours.

Architecture C — Context Enrichment Layer (CEL) Centric

One or more CEL platforms sit between the orchestrator and all data systems. For structured warehouse data, a Structured CEL platform (e.g. Rig) owns schema grounding, query validation, sandbox execution, and workflow automation. For semi-structured knowledge corpora, a Knowledge CEL platform (e.g. Zaimler) builds domain ontologies and governs knowledge retrieval. Database vendors and knowledge stores are reduced to managed data providers. Best for: complex regulatory environments, multiple database and knowledge systems, cross-vendor federated agent access. Trade-off: Adds dependency on newer platform categories; however, provides the most comprehensive governance surface area — and the correct CEL type must be matched to the data type it governs.

Section Implications
  • Standardise on MCP as the mandatory integration protocol for all new agent-to-database connections — it provides the schema validation and contract enforcement that raw SDK connectors lack.
  • Implement "DB as Policy Store" for any agent operating in a regulated domain — externalising rules to a database makes them auditable and changeable without redeploying agent code.
  • Never allow agents to impersonate human users in database sessions — this collapses audit trail integrity and violates separation of concerns that regulators increasingly expect.
Section 04

Production Realities: What
Practitioners Actually Face

"The demos look incredible. The prototypes feel magical. But getting these systems to actually work at scale, in production, with real users and real stakes — that is a completely different game."

The gap between controlled demonstration and production reliability is the defining challenge of agentic AI in 2025. Out of 1,837 surveyed organisations, only 95 reported having AI agents live in production. Even within that small group, most teams are still early in capability, control, and transparency — still struggling to understand when their agents are right, wrong, or uncertain. Understanding why is essential for any architecture decision.

The NL2SQL Reality Gap

"NL2SQL is a solved problem… Not!"
Floratou et al., Microsoft Research — title of CIDR 2024 paper that became one of the most-cited practitioner references of 2025

Empirical studies on real-world production databases indicate that state-of-the-art NL2SQL systems achieve limited execution accuracy on realistic business queries. More critically, when queries are ambiguous or unanswerable, models tend to force an answer — producing plausible but semantically incorrect SQL, executing without error while returning wrong results.

On well-formed academic benchmarks, NL2SQL accuracy exceeds 88%. In production enterprise environments — characterised by ambiguous column names, inconsistent documentation, million-table data lakes, and overlapping schemas — the picture is fundamentally different. LinkedIn's internal Text-to-SQL chatbot, serving 300+ weekly users and featuring a mature knowledge graph for schema grounding, achieves only 53% correctness on its own internal benchmark. This is not a poor implementation — it is a state-of-the-art one.

Even the best data teams often do not have clear documentation about tables, columns, and metadata. With the rise of ELT where data is simply dumped into the warehouse from various sources and transformed on query, table and column names may be the only information available to the engine at configuration time. Schema grounding — providing LLMs with enriched, business-meaningful metadata — is therefore not a nice-to-have feature. It is the primary determinant of NL2SQL production quality.

The Three Production Barriers

The Three Production Barriers

  • Reliability requirements — A 5% error rate is acceptable for a chatbot. It is catastrophic for agents that update databases or make automated decisions. One corrupted database entry can halt operations.
  • Integration complexity — Building a demo agent takes days. Integrating it with Oracle, Salesforce, legacy databases, security protocols, and compliance requirements often exceeds the expected business value. Technical debt kills most pilots here.
  • Costs at scale — Token usage accumulates rapidly when agents make multiple LLM calls per task. Several organisations discovered scaling successful pilots to full production volume would cost more than their entire contact centre budget.
Incident Record — July 2025 (Replit)

An AI agent deleted a production database containing 1,200+ records despite explicit instructions for "code and action freeze." The problem was not model intelligence. The problem was over-permissioning at the protocol level. Proper OAuth scopes might have prevented this. Instead: chaos. This incident captures the single most important architectural lesson: agent-to-database access governance is not a policy consideration — it is a system design requirement. Permissioning must be enforced at the protocol and database layer, not in prompts.

Context Engineering: The Real Work

The shift from "prompt engineering" to "context engineering" has emerged as the clearest dividing line between teams that ship reliable LLM systems and those still wrestling with inconsistent results. Dropbox uses the term to describe the architecture required to prevent "analysis paralysis" in their AI assistant. Anthropic's engineering team defines it as the management of everything that goes into the context window: system prompts, tool definitions, conversation history, and retrieval strategy.

Industry Convergence: The "Context Engine" as the New Foundation Layer

The broader industry is converging on a structural insight: "How to upgrade the technically-focused Retrieval Engine into a Context Engine is becoming the key determinant for whether AI Agents can achieve scalable, cost-effective enterprise adoption" (RAGFlow year-end review, 2025). Investment firm Theory Ventures was among the first to systematically elaborate the "Context Platform" concept, identifying retrieval quality as the fundamental bottleneck in LLM applications. The practical implication: platforms like Rig (structured data) and Zaimler (knowledge corpora) are not peripheral middleware — they are the emerging foundational substrate for enterprise AI agent reliability. The database layer provides the data; the Context Engine makes that data intelligible to agents.

For database practitioners, context engineering translates directly to three investments: enriched schema metadata (business glossaries, column descriptions, example values); intelligent table selection for large data lakes (not all tables should be exposed to all agents); and query result validation before surfacing to users or downstream processes.

The Cost Reality

General-purpose data agents incur 178–1,015 seconds of inference time per query, and multi-agent or reflection setups may require up to 31.1 external model calls without commensurate accuracy gains. Cost management is not an afterthought — it is a first-class architectural constraint.

Essential Cost-Control Techniques
  • Semantic caching — return cached results for semantically equivalent (not just syntactically identical) queries
  • Query result reuse — shared result stores across concurrent agent sessions
  • Intelligent model routing — smaller models for classification tasks, frontier models reserved for complex multi-step reasoning only
Practitioner Voice
Chief Data & Transformation Officer · Major Asian Bank
"Now with agentic AI coming into play and putting runtimes into workflows, it gives you a whole different set of possibilities for day-to-day work. But the tech is the easiest bit. The processes, the structure, the people, the culture — that's the hard bit. Most times people think: we've got this beautiful technology, let's give it to everybody and things will get solved. It doesn't work like that. You'll need to drive an organisation-wide transformation and cultural shifts."
Section Implications
  • Treat NL2SQL accuracy as an engineering problem requiring schema grounding, not a model selection problem. No frontier model overcomes poorly documented schemas.
  • Build cost models for every agent deployment before committing to production — token consumption at scale is frequently the surprise that kills otherwise successful pilots.
  • Implement deterministic validation gates (query linting, EXPLAIN-based performance checks, row-count sanity checks) around every LLM-generated query before execution against production systems.
  • Move safety logic out of prompts and into infrastructure — prompt-based guardrails fail under adversarial conditions; database-layer enforcement does not.

Ports & Logistics (PSA / SATS)
"How do we build agents that can sense disruption before it strikes — across real-time AIS, weather, and global news data?"

From Reactive to Predictive: PSA and A*STAR IHPC combine AI, simulation, and real-world port data to detect early warning signals from AIS feeds, weather systems, and global news — shifting from reactive incident response to proactive disruption prevention.

Architecture Implication: For agents, this is a middle-layer challenge: exposing consistent, schema-grounded interfaces across heterogeneous operational systems without creating governance blind spots at system boundaries. ✈️
Aviation & Transport (SIA / SMRT)
"How do we use agents for safety-critical operations where a data error is not merely a business cost?"

Highest-Stakes Environment: SIA and SMRT operate in domains where data errors translate directly to safety, regulatory, and reputational risk. The constraint is not insufficient data — it is the opposite: multiple overlapping, authoritative systems (flight ops, maintenance, crew scheduling, passenger services, financial settlement), each with strict data retention and audit requirements.

Critical Design Rule: Agents must operate with explicit read/write boundaries enforced at the database layer, not in application code. Safety thresholds, maintenance intervals, and regulatory limits must be stored in version-controlled database tables (DB-as-Policy-Store) — not in prompts that can drift or be manipulated.

High-Value Use Case — ST Engineering MRO: Predictive maintenance agents traversing sensor data, parts inventory, and regulatory compliance records simultaneously represent one of the highest ROI agent workloads in this sector — with the governance architecture above as a prerequisite.

📡
Telecom & Tech (Singtel / ST Telemedia)
"How do we manage the data sovereignty and latency demands of agent infrastructure across multi-country, multi-cloud operations?"

The Data Sovereignty Dilemma: Singtel's footprint across Singapore, Australia, India, and Southeast Asia means any agentic architecture must accommodate data residency requirements that vary by country — and sometimes by data category within a country. Network telemetry and CDR records are simultaneously the most valuable AI input and the most heavily regulated data category.

Critical Practitioner Insight — Vectors & Privacy: Vector embeddings of customer records may qualify as personal data under GDPR or PDPA. They require the same governance controls as the underlying structured record. Organisations deploying cross-border semantic search must architect embedding storage with the same data residency discipline as their core databases.

Infrastructure Angle — ST Telemedia: As enterprises deploy agent infrastructure on cloud platforms, the network edge at which agent-to-database latency is determined becomes a competitive differentiator. Operators offering low-latency, jurisdictionally compliant data proximity will have structural advantages in agentic AI infrastructure.

🏥
Healthcare & Life Sciences (Manipal / Celltrion)
"How do we deploy agents over clinical data without creating HIPAA/PDPA exposure or clinical liability?"

Non-Negotiable Controls: PHI access by agents requires: dynamic masking enforced at the database layer; service-principal-based identity with clinical-role-specific privileges; and complete audit trails that satisfy regulatory inspection. These are architectural requirements, not compliance checkboxes.

The Schema Documentation Gap: Clinical databases (EHR, PACS, lab systems) are typically the least well-documented enterprise data systems. Column names like pat_id_t and enc_dt_val are commonplace — making schema grounding not merely helpful but essential for any NL2SQL agent operating in this domain.

Graph Databases for Clinical Reasoning: Adverse drug event networks, clinical pathway graphs, and multi-hop biologics supply chain traceability (relevant to Celltrion's manufacturing) are workloads where graph reasoning substantially outperforms flat vector retrieval. Manipal Health's multi-hospital India network creates a federated governance challenge analogous to Singtel's — each hospital may operate different EMR systems under different state-level data regulations.

🌱
Agri-Food & Trade (Olam Group)
"How do we build agents that can reason across commodity, logistics, and sustainability data in real time?"

The Multi-Model Data Challenge: Olam's value chain spans commodity trading to sustainability reporting — a near-perfect illustration of why the agent-native data layer cannot be a single-vendor solution:

  • Commodity trading & pricing → OLTP, sub-second latency, ACID consistency
  • Supply chain traceability & ESG → Graph traversal (farm-to-shelf provenance)
  • Demand forecasting & yield prediction → OLAP analytical depth, lakehouse
  • Farm IoT & sensor data → Time-series or streaming infrastructure

Agents that provide genuine business value — a supply disruption early-warning agent or a sustainability compliance agent — must federate across all four data types simultaneously. This is precisely the use case that middle-layer platforms are built to address.

Composite Case — Regional Bank: Middle-Layer Governance for Customer Service Agents

A mid-sized bank sought to deploy conversational agents across three core banking systems, a CRM, and a risk scoring database. Initial prototypes using direct LangChain connectors produced inconsistent results, intermittently exposed masked data to the wrong agent persona, and generated SQL queries triggering full-table scans on a production OLTP system during peak hours.

Deploying Zaimler for schema grounding and Gate Tech for ABAC policy enforcement resolved all three failure modes within 90 days. Schema grounding reduced NL-to-SQL errors by approximately 65%. Gate Tech's policy layer ensured the customer-facing agent persona could never access data fields above its defined privilege level, regardless of what the LLM generated. Query interception prevented all unindexed scans. The bank subsequently extended the architecture with Element's workflow primitives for back-office refund processing, adding explicit rollback primitives for automated write operations.

Composite Case — Port Operator: Reactive Agents for Supply Chain Exception Handling

A major port and logistics operator deployed a reactive agent architecture for exception resolution — part shortages, vessel delays, and customs holds — that previously took 4–6 hours to escalate manually. A CDC-based event stream from the TOS (Terminal Operating System) fed agents that automatically retrieved contextual data from the logistics database and supplier risk graph (Neo4j), generated resolution recommendations, and — for pre-approved exception types — triggered SAP workflow actions autonomously.

Mean time to resolution dropped to under 30 minutes for in-scope exception types. Human oversight was retained for exceptions requiring financial commitments above defined thresholds, enforced programmatically through the middle-layer's approval gate. The key architectural insight: the middle layer's DB-as-Policy-Store configuration was the governance mechanism — the agent did not decide when to escalate; the policy database did.

Open Source

"How is the open-source community shaping de facto standards that our architects should be building toward — not against?"

Enterprise technology history shows that proprietary architectures are frequently supplanted by open-source standards that achieve critical mass before incumbents can respond. The agent-native data layer is exhibiting the same dynamics. A vibrant open-source ecosystem is producing MCP servers, memory schemas, and GraphRAG patterns that are becoming reference implementations regardless of what any single vendor ships.

Open MCP Servers: The Integration Commodity

Community-maintained MCP servers now exist for PostgreSQL, MySQL, SQLite, Redis, Neo4j, and MongoDB, providing standardised tool schemas that LLM orchestrators can discover and invoke without custom connector development. Key patterns: read-only MCP servers with write restrictions enforced at the protocol level; schema-first definitions that expose entity descriptions (not just column names) to improve LLM query accuracy; and audit hook integrations that intercept every tool call and emit structured logs to SIEM systems.

GraphRAG: Structured Knowledge for Grounded Agents

GraphRAG — augmenting LLM retrieval with knowledge graph traversal — has moved from academic novelty to production deployment. The core insight is a fundamental limitation of flat vector search:

  • Vector similarity retrieval surfaces documents that are semantically close
  • Graph traversal surfaces entities and relationships that are structurally connected
  • Combined, they give agents contextual depth ("what does this document say?") and relational breadth ("how does this entity connect to others in our data model?")

For fraud investigation (financial services), adverse event chains (healthcare), and supplier network tracing (logistics), GraphRAG delivers order-of-magnitude improvements in reasoning accuracy over flat RAG alone.

Section 07

The Context Enrichment Layer:
Two Distinct Platform Families

"What exactly is the middle layer — and when practitioners look closely, is it actually one category or two?"

The middle-layer platform category emerged in response to a genuine architectural gap: native vendor agents are platform-specific, and generic orchestration frameworks are governance-light. But as the market has matured — and as vendors like Rig have released detailed capability documentation — it is clear that what practitioners call the "middle layer" actually comprises two distinct platform families, differentiated by the type of data they enrich context for.

Taxonomy: The Context Enrichment Layer — Two Families

The architectural function these platforms share is context enrichment — making data comprehensible and governable for AI agents. But they do so for fundamentally different data types:

  • Structured Data Context Platforms (e.g. Rig) — connect to relational databases and warehouses, auto-discover schema and business logic, validate queries before execution, and maintain context as schemas drift. Core problem solved: AI agents cannot understand raw database schemas without enriched, business-meaningful context.
  • Knowledge Context Platforms (e.g. Zaimler) — work with semi-structured knowledge corpora (claims data, policy documents, domain-specific text), building domain-specific ontologies automatically and providing NL-to-query over heterogeneous knowledge bases. Core problem solved: AI agents cannot reason reliably over complex multi-source knowledge without an ontological structure anchoring retrieval.

Both use MCP. Both sit between orchestrator and data. But their integration depth, context generation mechanism, and target use cases are materially different — complementary, not competing. In a mature enterprise AI stack, both may be present simultaneously.

Recommended Taxonomy for AI Tech Stack Design
Rather than grouping these under a single "middle layer" label, we recommend calling this tier the Context Enrichment Layer (CEL) — distinguishing within it between Structured CEL platforms (database-native, schema-grounded) and Knowledge CEL platforms (ontology-driven, corpus-grounded). This distinction matters for procurement, architecture design, and vendor evaluation. Note: the broader industry is converging on the term "Context Engine" or "Context Platform" (articulated by Theory Ventures and reinforced by RAGFlow's 2025 year-end review) to describe this tier — the CEL framing adds the structured/knowledge sub-classification that the industry-wide term does not yet make explicit.

What the Middle Layer Actually Does

Schema grounding is arguably the most impactful function. By providing LLMs with enriched, human-readable schema descriptions — including entity relationships, business glossary terms, and example values — middle-layer platforms dramatically reduce SQL generation errors. A bare database schema tells an LLM what columns exist; a grounded schema tells it what the business means by those columns. This is the difference between 40% and 80% query accuracy in production environments.

Query planning and validation intercepts LLM-generated SQL before execution, checking for structural correctness, privilege compliance, and performance safety (flagging full-table scans against production OLTP systems). This is a quality gate that neither the LLM nor the database natively provides.

Policy enforcement applies data access rules as a centralised middleware concern — which agents can see which data, under what conditions, with what masking. Rather than duplicating governance logic across multiple platforms, policy is declared once and enforced uniformly.

Observability and lineage provide complete visibility into every agent-initiated data interaction: what was asked, what was generated, what was executed, what was returned, and what action resulted.

Rig: Structured Data Context for AI Agents

Rig's positioning — clarified by their own documentation — is notably more specific than a generic "middle-layer" description implies. Rig is a structured data context engine: it connects to data warehouses (Snowflake, BigQuery, Redshift, Postgres), auto-discovers schema and business logic, maintains an enriched semantic layer that stays current as data drifts, and exposes this context to any downstream AI agent or orchestrator via MCP.

Three capabilities differentiate Rig from adjacent platforms. First, its auto-discovery and context maintenance — rather than requiring manual schema documentation (the bottleneck that kills most NL2SQL deployments in production), Rig scans the warehouse on day one and continuously updates its semantic understanding as tables, columns, and relationships drift. This addresses what Rig calls the core enterprise problem: "Data access for AI systems involves manual integration and backlog hell" — the 50+ ticket backlogs and 3–6 month waits that currently exist between business teams and the data they need for automation.

Second, its Agent Sandbox validates every LLM-generated query for syntax, permissions, and compute cost in approximately 10ms before it reaches the warehouse. Invalid or expensive queries never execute. Rig claims this reduces warehouse compute costs by 60–80% while maintaining a complete 100% audit trail — addressing a real practitioner pain point where unconstrained agent queries can trigger full-table scans or runaway compute costs at scale.

Third, its workflow automation layer lifts the platform beyond conversational analytics into operational automation. Trigger-based workflows ("if ROAS drops below 2.0, detect anomaly, break down by ad set, draft Slack summary, auto-pause underperforming campaigns") transform data agents from answering one-off questions to running repeatable, scheduled business processes. This is a critical distinction from most NL2SQL tools, which remain conversational-only.

The question raised by practitioners — whether Rig's schema auto-discovery involves a degree of ontology building from the database schema — is architecturally astute. Rig's context engine does more than surface column names: it infers metric formulas, domain terms, table join logic, and access control hierarchies into a structured semantic layer. This is functionally analogous to lightweight ontology construction, though applied to structured relational data rather than the unstructured knowledge corpora that dedicated ontology platforms address. Whether this constitutes "ontology" in the formal sense, or a more pragmatic semantic metadata layer, is a distinction worth watching as the platform matures.

Rig vs Zaimler: Complementary, Not Competing

The practitioner question — "would we group them together or separately?" — has a clear architectural answer: separately, but in the same tier. Both Rig and Zaimler occupy the Context Enrichment Layer. Both use MCP. Both reduce hallucinations by providing AI agents with richer context than raw data sources alone provide. But their complementarity is their strength:

Rig — Structured Data Context
  • Works with: relational DBs, warehouses, lakehouses
  • Core mechanism: schema auto-discovery + semantic enrichment
  • Key differentiator: query sandbox, drift detection, workflow automation
  • Primary use cases: RevOps, fraud investigation, churn analytics, ad spend optimisation
  • Goes live: day one (no manual mapping)
  • Best fit: teams drowning in data team backlogs and SQL requests
Zaimler — Knowledge Context
  • Works with: semi-structured corpora, claims data, policy documents
  • Core mechanism: domain-specific ontology auto-generation
  • Key differentiator: cross-source federation, NL-to-query over heterogeneous knowledge
  • Primary use cases: insurance claims, compliance queries, cross-source analytical agents
  • Goes live: after ontology validation with domain experts
  • Best fit: teams with complex knowledge domains that resist simple schema description

An enterprise deploying both a warehouse-backed RevOps agent (Rig) and a compliance knowledge agent over insurance policy corpora (Zaimler) is not duplicating its middle layer — it is correctly matching the right context enrichment mechanism to the right data type. This is the architecture discipline the "Context Enrichment Layer" framing makes visible.

Section 09

Signals to Watch &
Failure Patterns

01
Vendor Convergence on Middle Layer

Major DB vendors are beginning to acquire or OEM middle-layer capabilities. The strategic window for independent governance platforms may narrow as Oracle, Databricks, and Snowflake absorb this category.

02
Regulatory Crystallisation

EU AI Act implementation, MAS guidance on agentic AI in banking, and HIPAA interpretation for agentic systems are all active in 2025–2026. Organisations without agent identity and audit logging face mounting compliance exposure.

03
MCP / A2A Standardisation

MCP standardises how agents connect to external tools, databases, and APIs — this transforms custom integration work into plug-and-play connectivity. If it achieves TCP/IP-style universality, basic connector infrastructure will be commoditised.

04
Graph-Vector Convergence

The architectural separation between vector search and graph traversal is narrowing. Neo4j, Weaviate, and Amazon Neptune are shipping combined capabilities. If this converges into a de facto hybrid store, multi-system retrieval architectures will simplify considerably.

05
In-Database LLM Inference

Oracle, SAP, and IBM are embedding lightweight LLM inference directly within database engines — enabling reasoning over data without data movement. This could alter the latency and data sovereignty profile of in-DB agent architectures materially.

06
Agent Memory Standardisation

The absence of a universal agent memory schema creates fragmentation. OpenAI, Anthropic, and the open-source community are advancing memory frameworks. The first to achieve enterprise adoption will set a standard with significant lock-in implications.

07
CEL Category Maturation

The Context Enrichment Layer is bifurcating into Structured CEL (database-native, schema-grounded) and Knowledge CEL (ontology-driven, corpus-grounded). As both categories mature rapidly — Rig claiming day-one deployment vs months for manual approaches — watch for acquisition activity as database vendors seek to absorb CEL capabilities rather than build them.

Four Failure Patterns to Actively Avoid

Failure Pattern 1 — Automating Broken Processes

The most common failure pattern is not a technology failure — it is a process failure at machine speed. Agents deployed over poorly understood or inconsistently executed business processes do not fix the process; they execute it with higher velocity and lower human oversight. Rigorous process mining and workflow documentation must precede, not follow, agent deployment. Many agentic AI implementations are failing because current enterprise data architectures, built around ETL processes and data warehouses, create friction for agent deployment — most organisational data is not positioned to be consumed by agents that need to understand business context and make decisions.

Failure Pattern 2 — Agent Washing

Products with basic chatbot or NL2SQL capabilities are being repositioned as "autonomous agents" without the multi-step planning, tool-use, and self-correction capabilities that the term implies. CTOs and CDOs should demand explicit demonstrations of multi-step reasoning, tool chaining, and error recovery — not just NL query interfaces — before classifying a product as agentic infrastructure. While 30% of organisations are exploring agentic options and 38% are piloting solutions, only 14% have solutions that are ready to be deployed and a mere 11% are actively using these systems in production.

Failure Pattern 3 — PoC Proliferation Without Governance Foundation

Running 10–15 agent PoCs simultaneously, with minimal governance infrastructure and no shared architectural standards, is producing technical debt at scale. Each PoC tends to implement its own connector pattern, memory management approach, and security model — creating a fragmentation problem that compounds with each additional deployment. Governance and architectural standards should be established at the first PoC, not retrofitted after the tenth. Most implementations lack defined failure modes, hallucination handlers, graceful degradation, and human-in-the-loop safeguards. When errors occur, the system simply keeps going. These are not exotic bugs — they are predictable consequences of treating agents as features instead of systems.

Failure Pattern 4 — Shadow Agents and Data Perimeter Drift

As agent-building tools become more accessible, individual business units and developers are deploying agents against enterprise data systems without architecture review or security approval — often using personal API keys and shared database credentials. This creates data perimeter drift that is difficult to detect and expensive to remediate. Organisations should implement a formal agent registry and require all agents accessing enterprise data systems to be declared, governed, and monitored — regardless of their originating team. 70% of regulated enterprises rebuild their AI agent stack every three months or faster, underscoring how unstable production environments remain. A fragmented PoC landscape accelerates this instability.

Closing Perspective

The organisations that will differentiate over the next three years are not those with the most sophisticated agents or the most advanced LLMs. They are the organisations that have built the governance, observability, and architectural discipline to deploy agents reliably, accountably, and at enterprise scale. The agent-native data layer is the infrastructure of that future. DBS Bank's S$1 billion in measured AI value is not a story about model capability — it is a story about 10 years of data infrastructure investment, governance discipline, and incremental capability building. The decisions made in the next 12–18 months will determine whether the agent data layer becomes a competitive accelerant or a governance liability. The evidence from practitioners is clear: start with governance, not with the agent.

Deep Dive: BigQuery Data Agents vs Rig — When Each Wins

The Rig deck makes a direct comparison that is worth understanding. BigQuery Data Agents (Google's native conversational analytics) require manual context setup — you write field descriptions, glossary terms, and verified queries yourself. Rig's context engine auto-discovers these. This creates a meaningful tradeoff:

Choose BigQuery Data Agents when:
  • Your data is entirely in BigQuery and you plan to stay there
  • Your schemas are relatively clean and well-documented already
  • You want conversational analytics only (no workflow automation)
  • You have a data team available to write and maintain context
  • Vodafone's migration to BigQuery showed 90% reduction in ETL migration time — for teams already mid-migration, native tooling has real integration advantages
Choose Rig when:
  • Your schemas are complex, poorly documented, or changing frequently
  • You need agents to span multiple data sources (not just BigQuery)
  • You need workflow automation, not just Q&A
  • You want query cost control (sandbox prevents runaway warehouse spend)
  • Your data team is the bottleneck — Rig removes the manual context mapping that creates 3–6 month backlogs

Note: These are not mutually exclusive. A mature architecture may use BigQuery Data Agents for ML and data engineering workflows while using Rig as the CEL layer for operational agent automation across a broader data estate.

04
Build the Pilot — Governance First
Week 3–4 · Engineering + Data team

The pilot is not a proof-of-concept. It is a production rehearsal. Every architectural decision made in the pilot will be amplified at scale. Build as if you are going to production.

□
Create a dedicated service principal for the agent with minimum-privilege database credentials. Never share credentials with human user accounts.
□
Implement schema grounding before any NL2SQL query runs. Document at minimum: table purpose, column descriptions for all columns the agent will touch, known ambiguous terms, and 3–5 example verified queries.
□
Set up structured audit logging from day one. Every query, every result, every action should log: agent ID, timestamp, natural language input, generated SQL, rows returned, and downstream action taken.
□
Define the rollback protocol before any write-capable agent is tested. What is the transaction boundary? Who can halt the agent? What is the recovery process if a write is incorrect?
□
Run the pilot read-only first. Validate accuracy and governance against your baseline metric before enabling any write or action capabilities.
□
Measure against your baseline. If the agent was supposed to reduce fraud investigation time from 3 hours to 30 minutes, measure it now. If you cannot demonstrate improvement in the pilot, do not proceed to production.
05
Scale — Expand with Discipline
Month 2+ · Cross-functional

A successful pilot earns the right to scale — not the right to rush. The failure pattern is clear: organisations that succeed in the pilot then deploy 10 agents simultaneously without shared governance infrastructure, creating fragmentation that is expensive to remediate.

Scaling Do's
  • Establish an agent registry — every agent accessing enterprise data is declared, documented, and monitored
  • Standardise on MCP as the integration protocol for all new agent-to-data connections
  • Apply the graduated autonomy model consistently: read → semi-autonomous → write-capable, with formal review gates at each step
  • Share schema grounding documentation across agents — don't rebuild context for every new use case
  • Measure and publish business impact (time saved, cost reduced, accuracy improved) to build institutional confidence
Scaling Don'ts
  • Don't allow shadow agents — business units deploying agents without architecture review or security approval
  • Don't deploy write-capable agents to regulated systems without board-level risk acknowledgement
  • Don't accumulate agent memory in unmanaged vector stores without TTL policies and PII purge mechanisms
  • Don't skip post-pilot governance review to "move faster" — the technical debt compounds rapidly
  • Don't treat a successful pilot as proof that all related use cases are ready — each new data source requires its own governance assessment

The architecture described in this report is not a future aspiration — it is deployable today, with existing tools, by teams that are disciplined about starting with governance rather than ending with it. The organisations that will lead in the agentic era are those that start narrow, instrument everything, and scale deliberately.
