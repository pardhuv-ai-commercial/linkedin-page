01 — Diagram formats

What the community is actually using

Six formats dominate practitioner GitHub repos, engineering blogs, and Substack newsletters. Each occupies a distinct layer of the stack and serves a different audience.

Mermaid

Most used

Text-as-code diagrams that render natively inside GitHub, GitLab, Notion and Confluence. Flowcharts, sequence, C4, ERD — all from markdown. The community standard because it travels in pull requests.

C4 model

De facto standard

Four zoom levels — Context → Container → Component → Code — mapped to four different audiences. The architectural storytelling framework that scales from board slides to code review. Draw it in Mermaid or Structurizr DSL.

02 — Markdown file formats

What gets checked into repos

The practitioner community has converged on docs-as-code: every architecture artefact lives in a git repo as a markdown file, updated in pull requests alongside code.

ADR — Architecture Decision Record

Most adopted

One file per architectural decision. Lightweight but durable. Captures the why — context, options considered, decision made, consequences. Lives in /docs/decisions/.

Agent spec file (agent.md)

Emerging standard

New AI-era convention. One file per deployed agent: model, system prompt, tools, memory config, approval gates, observability hooks, rollback. Checked in alongside code — the agent's "passport".

The anti-pattern to avoid

The community has largely rejected Visio and heavy UML tooling for agentic systems. The core problem: if the diagram can't be updated in a pull request, it will go stale. Agentic architectures evolve on a weekly cadence — a diagram that lives outside the codebase becomes archaeology within a month. The other anti-pattern is the "everything connected to everything" spaghetti graph — technically accurate, but communicates nothing about intent or behavior. Pick one diagram per audience, keep it sparse, and let the prose do the rest.

05 — Guiding principles

Four rules the community has learned the hard way

01

One diagram per audience

Never try to serve all stakeholders with a single diagram. The C4 model exists precisely to solve this: four zoom levels, four audiences. A board-level context diagram and a developer component diagram are different artefacts that should never be merged.

02

Diagram source must live in git

Any diagram whose source is a binary file (Visio, Lucidchart, Figma) will go stale. Mermaid, PlantUML and Structurizr DSL are text — they diff, merge, and get reviewed in PRs. For agentic systems that change weekly, this is non-negotiable.

03

Document the why, not the what

ADRs are the highest-leverage documentation investment. A developer six months from now doesn't need a diagram of what was built — the code does that. They need to know why LangGraph was chosen over CrewAI, and what trade-offs were accepted.

04

Agents read docs too — write for them

CLAUDE.md and AGENTS.md files are a new documentation layer that didn't exist two years ago. Mermaid flowcharts embedded in these files act as execution contracts — the agent reads the decision tree before writing a single line of code.
