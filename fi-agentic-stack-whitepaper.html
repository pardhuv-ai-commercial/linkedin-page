01 — The problem
● Real operational friction — observed in production

Why multi-party AI support fails

In regulated financial environments, every layer of an AI stack is owned by a different party. The product vendor wrote the model. The system integrator manages day-to-day support. The infrastructure provider runs the hardware. The financial institution owns the data and the regulatory obligation. This structure creates four compounding frictions that manual workflows cannot resolve at speed.

01
Physical + Security Isolation — "The Two-Screen Problem"
Air-gapped production environments prevent vendors from viewing live logs. Support engineers physically sit at two laptops — one with production data, one with vendor access — manually transcribing stack traces, model outputs, and telemetry between screens. Each transcription introduces error. Each error adds hours to MTTR. Just-In-Time (JIT) access approval workflows add a 2–4 hour "access dead zone" before a vendor can even begin diagnosing the issue.
IAM / JIT Air-gap Manual transcription
"We spent more time getting access than fixing the issue. By the time the vendor could log in, the model had already auto-recovered — but nobody knew why it broke." — Production engineer, Asian financial institution (Reddit r/mlops, 2025)
02
Cross-Platform Context Decay — "The Three-Ticket Problem"
Financial institutions run on MS Teams or Slack. AI/ML vendors use enterprise email and Jira. System integrators use ITSM platforms (ServiceNow, Remedy). When a fix is identified in one system, it must be manually re-entered in all others — with each re-entry losing nuance, context, and linkage. The result: the audit trail that regulators require is scattered across three disconnected systems with no shared incident ID. Reconstructing a timeline for an examiner takes days.
Audit trail Platform fragmentation
"After an incident our MAS examiner asked for a timeline. I had to stitch together three Slack threads, two email chains and a ServiceNow ticket. It took a week." — Compliance officer, Singapore bank (LinkedIn, 2025)
03
Expertise Dilution — "The Knowledge Gradient Problem"
Frontline support (L1/L2) at system integrators lacks the deep ML domain knowledge to diagnose model drift, false-positive threshold failures, or typology mismatch issues. Without direct access to the AI vendor's internal knowledge base — model versioning history, AFC typology library, known failure patterns — they apply generic IT fixes to highly specialised AI problems. This resolves symptoms while leaving root causes embedded in the model configuration or training data pipeline.
Model drift Generic fixes Knowledge silo
"L1 kept restarting the service. The actual issue was a data schema change in the upstream feed that was silently corrupting feature vectors. You can't find that without knowing the model." — ML engineer, AML vendor (HackerNews, 2025)
04
Explainability Debt — "The Black-Box Regulator Problem"
95% of organisations have experienced at least one AI incident (Infosys 2025), yet only 2% have adequate guardrails. In AML, up to 95% of alerts are false positives — and regulators increasingly require explainable, auditable reasoning behind every suppressed alert. Traditional ML pipelines produce no native explanation trail. When a model suppresses a suspicious transaction, the institution must demonstrate why — but the decision chain lives in a neural network with no queryable audit log. Agentic AI compounds this: autonomous multi-step decisions are even harder to reconstruct post-hoc.
Explainability Regulatory risk Audit reconstruction
"The examiner's exact words: 'Show me why your model cleared this transaction.' We couldn't. That was a $3M fine." — CCO, European bank (ACAMS conference notes, 2025)
The structural diagnosis

These are not skills problems or tooling problems. They are architecture problems. Each party owns one layer of the solution stack, but there is no integration layer connecting them — no shared context, no automated handoff, no unified audit trail. Agentic AI is precisely that missing integration layer — but only if architected correctly. Deployed naively, it adds a fifth layer of complexity to the existing four.

02 — Deep stack
● Component-level architecture · file/folder depth

The full AI stack — component by component

Seven layers, each expanded to the real components, files, connectors, and models that comprise it in production. Read top-down for data flow. Read by colour for ownership. Every component listed here has a known failure mode documented in Section 4.

L0 Regulatory & compliance boundary — wraps all layers
Financial institution · regulator-mandated
Regulatory controls (always-on, enforced before any agent action)
IAM
JIT Access Broker
PAM platform · CyberArk / BeyondTrust
Programmatic JIT request API. Agent pre-stages access request at T+0; human approves within 10 min vs. legacy 4h. All grants time-boxed and scoped to minimum required permissions.
Audit
Immutable Audit Ledger
SIEM · Splunk / IBM QRadar + hash-chain signing
Every agent action, tool call, LLM prompt+response, approval, and cross-party message captured with ML-DSA-44 post-quantum signatures. MAS-exportable. Cannot be deleted by any party — including the institution itself.
Policy
Policy Engine + Kill Switch
OPA (Open Policy Agent) · agent-guardrails lib
Hard-coded policy limits enforced at infrastructure level — not in agent code. Per-agent token budgets, cost caps, tool permission scopes. Kill switch: any human can halt all agents institution-wide within 30 seconds.
Data residency enforcement
DLP
PII / Data Scrubber
Presidio (Microsoft) · custom NER model
Strips PII, account numbers, and transaction amounts from all data before it crosses any party boundary. Produces anonymised "Diagnostic Packages" — the only artefact allowed to leave the institution's boundary.
HITL
Human-in-the-Loop Gate
Approval workflow · LangGraph interrupt() · Teams / Slack webhook
Configurable checkpoints pause agent execution and route to a named human approver before any write action, vendor data share, or change request execution. Approval, rejection, or modification — all recorded with diff.
Risk
Autonomy Level Registry
YAML config · /config/agent-autonomy.yaml
Explicitly classifies each agent as: Observe-only / Recommend / Act-with-approval / Act-autonomously. Most agents should remain at Recommend level. Act-autonomously requires board-level sign-off and 6-month production track record.

Primary LLM — reasoning, tool-calling, planning
Hosted API
Claude Sonnet / Opus
anthropic.Anthropic() · model="claude-sonnet-4-5" · max_tokens=4096
Primary reasoning model. Strengths: long context (200k tokens), strong tool-calling, structured output via JSON mode. Used for: diagnosis synthesis, root cause analysis, CR drafting. Cost routing: Sonnet for heavy reasoning, Haiku for classification.
Fallback
GPT-4o / Gemini 1.5 Pro
openai.OpenAI() · model="gpt-4o" · response_format=json
Failover model if primary API is degraded. Circuit breaker pattern: if primary LLM returns 3 consecutive 5xx errors, orchestrator automatically routes to fallback. Fallback must be pre-approved for data residency compliance.
Self-hosted
Llama 3.3 70B (on-prem option)
vLLM server · /v1/chat/completions · GPU: 2× H100 min
Required when MAS data residency prohibits sending any data — even anonymised — to external APIs. Deployed via vLLM on HPE ProLiant with NVIDIA H100 cluster. Latency ≈3× higher but zero data egress. FI must manage model versioning and drift.
Specialised models — domain-specific tasks
Embedding
text-embedding-3-large / BGE-M3
embed_dim=3072 · batch_size=512 · pgvector index
Converts typology documents, AML policy text, and incident reports into vector embeddings for RAG retrieval. BGE-M3 for on-prem deployments; text-embedding-3-large for cloud. Embeddings re-indexed nightly as policy docs update.
Classification
Incident Classifier (fine-tuned)
bert-base · /models/incident-clf-v3.safetensors · labels: [drift, schema, infra, config]
Fine-tuned BERT classifier that routes incoming incidents to the correct specialist agent. Trained on 18 months of historical incidents. Outputs: incident_type, confidence, recommended_agent. Retrained quarterly or when precision drops below 0.85.
Structured
PydanticAI Output Parser
pydantic.BaseModel · /schemas/diagnostic_package.py
Enforces typed output from every LLM call. If LLM output doesn't conform to schema, retries up to 3 times with corrective prompt before raising to HITL. Prevents agent runtime errors caused by malformed JSON — the most common production failure mode.

↓ orchestration instructions · state transitions
L2 Agent orchestration layer — multi-agent mesh
Shared · financial institution leads governance
Orchestration runtime — LangGraph
Orchestrator
LangGraph StateGraph
from langgraph.graph import StateGraph · checkpointer=PostgresSaver
Defines agent workflow as a directed cyclic graph. Nodes = agent actions. Edges = conditional transitions. Supports loops (retry on failure), parallel fan-out (simultaneous CR creation), and interrupt/resume for HITL gates. State persists in Postgres — survives process restarts.
State
Shared Agent State
TypedDict · /agents/state.py · fields: incident_id, diagnostic_pkg, fix_rec, cr_draft, approvals[]
Single typed state object passed between all agents. Contains: incident ID (shared across all platforms), diagnostic package, specialist recommendation, drafted CR, and approval chain. Immutable once written — each agent appends, never overwrites.
Memory
Checkpointer + Thread Memory
PostgresSaver · thread_id=incident_id · ttl=90d
LangGraph checkpointer persists full workflow state to Postgres after every node execution. Thread ID = incident ID. If process crashes mid-workflow, execution resumes from last checkpoint. TTL 90 days for compliance retention, then archived to cold storage.
The four agents — scoped per party boundary
Phase 1
Sentry Agent
/agents/sentry.py · tools: [hpe_telemetry_read, apm_logs_read, pii_scrub] · boundary: FI only
Runs inside FI's air-gap boundary. Read-only. Collects HPE hardware telemetry (CPU/GPU utilisation, memory pressure, network I/O), APM application logs (AppDynamics/Dynatrace), and AML model inference logs. Applies PII scrubber before producing Diagnostic Package. No data leaves boundary.
Phase 2
Specialist Agent
/agents/specialist.py · tools: [typology_search, model_history_read, fix_db_query] · boundary: vendor env
Runs in AI/ML vendor's environment. Receives anonymised Diagnostic Package via MCP. Queries internal typology library, model version history, and historical fix database to identify root cause and recommended remediation. Returns structured FixRecommendation with confidence score and evidence citations.
Phase 2
Investigator Agent
/agents/investigator.py · tools: [cr_template_fetch, env_context_read, servicenow_cr_create] · boundary: SI env
Runs in system integrator environment. Enriches fix recommendation with FI environment context (model version deployed, last config change, current alert volume). Drafts pre-filled Change Request using ServiceNow CR template. Validates required fields against change management policy before submission.
Phase 2
Coordinator Agent
/agents/coordinator.py · tools: [teams_message, servicenow_cr_link, email_send, jit_prestage, audit_write] · boundary: integration layer
Orchestrates cross-party communication. Simultaneously: (1) sends HITL approval request to FI compliance analyst via Teams/Slack, (2) links CR in ServiceNow with shared incident ID, (3) sends summary email to vendor. All three actions carry identical incident_id — creating a linked audit chain across all platforms. Pre-stages JIT access request at T+0 before it's needed.
Optional
Evaluator Agent
/agents/evaluator.py · tools: [metric_read, baseline_compare, alert_fn_compute] · runs post-fix
Post-resolution agent. Monitors model performance metrics for 72h after fix deployment. Compares alert volume, false positive rate, and model confidence against pre-incident baseline. Generates automated "Incident Closed" report with before/after metrics. Triggers re-escalation if metrics don't recover within SLA.
Guard
Prompt Injection Guard
rebuff / llm-guard · /middleware/injection_guard.py · blocks: jailbreak, data exfil, tool abuse
Every agent prompt passes through injection guard before LLM call. Detects: prompt injection via tool outputs (malicious log entries), jailbreak attempts, attempts to access tools outside declared scope. Known attack vector: adversarial AML alert narrative designed to hijack Investigator agent. Fail-closed — if guard is uncertain, routes to HITL.

↓ all tool calls route through MCP gateway
L3 MCP gateway — universal integration protocol
Shared · FI enforces permissions
MCP server — tool registry and execution
Protocol
MCP Server
@modelcontextprotocol/server-node · JSON-RPC 2.0 · stdio/SSE transport
Each tool is registered once as an MCP tool definition with: name, description, inputSchema (JSON Schema), and handler function. Any agent with permission can discover and invoke any registered tool without custom integration code. Before MCP: N agents × M tools = N×M integrations. With MCP: M tool registrations + N permission grants.
Auth
OAuth 2.0 / mTLS
OAuth 2.0 PKCE · mTLS client certs · token TTL=15min
Every agent authenticates to MCP gateway with a short-lived OAuth token (15 min TTL). Write-capable tools additionally require mTLS client certificate pinned to the agent's deployment identity. All token grants logged to audit ledger. Token refresh is automatic but revocation propagates within 60 seconds.
Enforce
OPA Permission Engine
openpolicyagent.org · /policies/tool-permissions.rego · deny-by-default
Deny-by-default. Every tool call evaluated against Rego policy: agent_id + tool_name + action_type must match an explicit allow rule. Sentry Agent: [hpe_telemetry_read, apm_logs_read, pii_scrub]. Coordinator: [teams_message, servicenow_cr_link, jit_prestage]. No agent can invoke a tool not in its declared allow-list, even if it constructs a valid MCP call.
Registered MCP tools — production tool catalog
Read · HPE
hpe_telemetry_read
HPE iLO REST API · /redfish/v1/Systems · metrics: cpu_util, gpu_mem, net_io
Pulls hardware telemetry for specified time window. Rate-limited: max 60 req/min per agent. Returns structured JSON: {timestamp, node_id, cpu_pct, gpu_mem_gb, net_rx_mbps, net_tx_mbps, anomaly_score}.
Read · APM
apm_logs_read
AppDynamics REST API / Splunk HEC · index=aml_prod · time_range=last_2h
Pulls structured application logs: model inference latency, alert volume per typology, false positive counts, error rates. PII scrubber applied automatically before returning. Logs older than 90 min require JIT access approval to retrieve.
Read · Vendor
typology_search
Vendor RAG API · POST /v1/typology/search · query: incident description · top_k=5
Semantic search over vendor's AFC Ecosystem typology library. Returns relevant typology documents with confidence scores. Input: anonymised diagnostic package text. Output: [{typology_id, name, description, known_fixes[], model_versions_affected[]}].
Write · ITSM
servicenow_cr_create
ServiceNow Table API · POST /api/now/table/change_request · requires HITL approval token
Creates Change Request in ServiceNow. Pre-fills: title, description, root_cause, fix_steps, risk_level, affected_systems, rollback_plan. Requires valid HITL approval token in request body — cannot be called without human approval. Links CR to shared incident_id for cross-platform audit.
Write · Comms
teams_message
Microsoft Graph API · /v1.0/teams/{id}/channels/{id}/messages · Adaptive Card payload
Posts structured approval request to designated Teams channel. Uses Adaptive Card format with approve/reject/modify action buttons. Button click generates signed approval token consumed by servicenow_cr_create. Message includes: incident summary, risk assessment, proposed fix, estimated impact, rollback plan.
Write · IAM
jit_prestage
PAM REST API · POST /v1/access-request · scope: read_prod_logs · ttl=4h · requester: vendor_agent_id
Pre-stages JIT access request at incident T+0 before vendor access is needed. Human still approves — but request is ready immediately. Reduces access dead zone from 2–4h to 8–12 min (approval time only). Auto-expires after TTL. All grants hashed and linked to incident_id in audit ledger.

L4 Memory & knowledge layer
Shared · data classified per residency rules
Three-tier memory architecture
Tier 1 · in-flight
Context Window
128k–200k tokens · within LangGraph node · cleared per incident
Active working memory for current task. Contains: conversation history with tools, intermediate reasoning steps, tool results, partial state. Cleared after each incident closes. Token usage monitored per agent — when context exceeds 80% capacity, summarisation agent compresses and offloads to episodic store.
Tier 2 · retrieval
Vector Store (RAG)
pgvector on PostgreSQL · 1536-dim · cosine similarity · IVFFlat index · ANN k=5
Semantic search over: AML typology library, regulatory policy docs (MAS Notice 626, FATF guidelines), historical incident reports, model change logs, and vendor fix archives. Updated nightly. Query: anonymised incident description → top-5 relevant context chunks injected into agent prompt. Documents expire after regulatory retention period.
Tier 3 · persistent
Episodic Store
Mem0 / Redis + TTL=180d · key: agent_id + entity_type + entity_id
Cross-incident memory: recurring incident patterns, vendor fix effectiveness scores, analyst preferences, model performance baselines. Example: "Typology X has been falsely triggering on payment type Y for 3 months — last 4 incidents all required the same config change." Agents query this before proposing new fixes. Retrained quarterly.
Knowledge sources — what agents can query
Regulatory
Policy Document Store
MAS Notice 626 · FATF Rec 10/16 · FAFT guidance PDFs → chunked + embedded
Regulatory guidance documents chunked at paragraph level and embedded. Agents query when drafting CR justifications or HITL approval context to cite specific regulatory requirements. Updated when regulators issue new guidance — automated ingestion pipeline with human review gate.
Vendor KB
AFC Typology Library
Vendor RAG endpoint · /v1/typology/ · updated via federated learning contributions
Proprietary typology library containing known financial crime patterns, model scenario configurations, and fix histories. Queried by Specialist Agent via MCP (anonymised package only — no raw FI data sent). Vendor maintains ownership. FI can contribute anonymised pattern data back via federated learning.
Internal
Historical Incident DB
PostgreSQL · incidents table · indexed by: typology_id, model_version, fix_applied, resolved_bool
3-year history of all incidents, applied fixes, resolution times, and post-fix metric deltas. Agents query to find: "What was applied last time this typology misfired? Did it work?" Prevents re-applying fixes known to be ineffective. Ingested from legacy ITSM on initial setup.

04 — Governance
● Regulatory alignment · APAC + global frameworks

How the architecture satisfies regulators

Regulators will ask three questions about any agentic AI system: "How do you know what your AI agent did? Why did it do it? Should it have?" Every design choice in this architecture is aimed at answering all three with specific, documented evidence. Only 2% of organisations currently have adequate AI guardrails (Infosys 2025). This section addresses that gap.

Pillar 1 · Data residency
Zero raw data crosses boundaries
  • All transaction data, customer records, and model artefacts remain within FI's geographic boundary
  • Presidio + custom NER scrubs PII from all outbound data before MCP tool invocation
  • Anonymised Diagnostic Package is the only artefact allowed to cross to vendor environment
  • OPA data-residency.rego policy blocks any tool call that would send Restricted-classified data outside boundary
  • For strictest MAS compliance: on-prem LLM (Llama 3.3 70B via vLLM) — zero data egress to external API
Pillar 2 · Audit trail
Every action reconstructable for examiners
  • Every agent action, LLM prompt+response, tool call, approval, and cross-party message written to immutable SIEM
  • Hash-chain signing (ML-DSA-44) makes audit records tamper-evident — institution itself cannot delete them
  • Shared incident_id links Teams thread + ServiceNow CR + vendor email + audit log into one reconstructable timeline
  • Regulators can request: full incident timeline, agent reasoning chain, human approval decisions, before/after metrics
  • Automated quarterly audit report generated by Coordinator Agent — reduces examiner prep from weeks to hours
Pillar 3 · Human oversight
Humans remain accountable for decisions
  • Autonomy level registry explicitly classifies every agent action — most remain at Recommend level
  • Act-with-approval required for: CR creation, vendor access grant, model parameter change, watchlist override
  • Act-autonomously reserved for: monitoring-only actions with zero write capability and 6-month track record
  • HITL gate with delegate chain: 2h escalation → 4h page on-call manager. No indefinite suspension.
  • Kill switch: any named human can halt all agents institution-wide within 30 seconds via PAM console
Pillar 4 · Explainability
Every recommendation cites its evidence
  • Specialist Agent structured output schema requires: root_cause, confidence_score, evidence_citations[], typology_id
  • Every HITL approval request includes: diagnostic package summary, recommended fix, typology reference, risk assessment, rollback plan
  • Evaluator Agent produces post-fix metrics report with before/after comparison — not just "it worked"
  • LLM call logs retained with full prompt + response — not just the output. Reasoning chain reconstructable.
  • Model decisions (alert suppression, risk scoring) use XAI layer — SHAP values logged per prediction

05 — Architecture decision records
● ADRs · docs-as-code · /docs/decisions/

Why the key decisions were made

Each decision below is an ADR that belongs in your repository. The "why not" is as important as the "why" — documenting rejected alternatives prevents re-litigating the same choices when team members change or vendors pitch alternatives.

ADR-001
Distributed agent deployment — each agent in its owner's boundary
Accepted
ContextMAS data residency prohibits raw financial data leaving Singapore or crossing to third-party environments uncontrolled. All-cloud deployment would violate this.
RejectedAll agents on shared cloud (data residency violation); all agents inside FI (vendor loses access to own knowledge base)
DecisionDistributed: Sentry in FI boundary; Specialist in vendor env; Investigator in SI env; context crosses only as anonymised packages via MCP.
ConsequenceMAS compliant. Latency overhead ~800ms per cross-boundary call (acceptable). Operational complexity: 4 deployment environments to manage.
ADR-002
LangGraph as primary orchestration framework
Accepted
ContextSupport workflows are non-linear: diagnosis loops, conditional escalation, HITL gates that pause indefinitely, parallel CR creation. Linear chains cannot model this.
RejectedCrewAI (role-based, poor branching); AutoGen (Microsoft stack dependency); custom state machine (maintenance burden); simple LangChain (no loop/interrupt support)
DecisionLangGraph with PostgresSaver checkpointer. Graph state machine supports loops, rollback, HITL interrupt/resume, and parallel fan-out natively.
ConsequenceFull workflow traceability. State survives process restarts. Debuggable via LangSmith. Learning curve for engineers unfamiliar with graph-based state machines.
ADR-003
MCP gateway as universal integration standard
Accepted
ContextWithout a standard, each agent–tool connection requires custom integration code: N agents × M tools = N×M integrations to maintain. At 5 agents and 8 tools = 40 bespoke connectors.
RejectedCustom REST adapters (N×M problem); direct SDK calls (no permission enforcement layer); A2A only (not yet stable for tool integration)
DecisionMCP mandatory for all tool registrations. OPA enforces permissions per agent. A2A adopted for future agent-to-agent direct communication.
ConsequencePrevents vendor lock-in. MCP now adopted by Anthropic, OpenAI, Microsoft, Google, Linux Foundation. Tool catalog versioned in YAML — schema drift detectable in CI.
ADR-004
On-prem LLM for strictest data residency vs. hosted API
Context-dependent
ContextSome MAS interpretations prohibit sending any data — even anonymised — to external LLM APIs. Other institutions use anonymised data + hosted Claude/GPT-4o. Interpretations vary.
Option AHosted API (Claude/GPT-4o): lower latency, better capability, no GPU ops burden. Requires MAS sign-off on anonymised data leaving FI boundary.
Option BOn-prem Llama 3.3 70B via vLLM: zero data egress, MAS compliant by default. Requires 2× H100 GPU, 3× higher latency, FI manages model versioning.
ConsequenceDecision must be made per institution based on MAS counsel's interpretation. Architecture supports both — llm_router.py switches between endpoints via config flag.
ADR-005
Deny-by-default OPA policy vs. allow-by-default agent code
Accepted
ContextAgent code can be modified by engineers. If permissions are encoded in agent code, a code change could silently grant an agent new capabilities outside its declared scope.
RejectedPermissions in agent system prompt (LLM can be instructed to ignore them); permissions in agent code (bypass via code change); no explicit permission model
DecisionOPA deny-by-default enforced at MCP gateway infrastructure level. Even if agent code changes, OPA blocks any tool call not in the explicit allow-list. Code changes cannot grant new permissions.
ConsequencePermission changes require OPA policy file update (in Git, requires PR review). Stronger security posture. Engineers must know OPA Rego to add new tool permissions.
ADR-006
PydanticAI structured output vs. free-text LLM responses
Accepted
ContextLLMs occasionally produce malformed JSON, skip required fields, or return explanatory prose instead of structured data. This causes silent agent runtime errors — the most common production failure mode observed in AML deployments.
RejectedFree-text parsing (fragile regex); manual JSON extraction (error-prone); trusting model to always return valid JSON (it doesn't)
DecisionPydanticAI schema enforcement on every LLM call. Invalid output → retry up to 3× with corrective prompt → HITL if still failing. Schemas version-controlled in /models/schemas/.
ConsequenceEliminates silent runtime failures. Schema acts as contract between agents. Adds ~200ms latency on retry cases. Retry rate monitored — sustained retry rate > 5% triggers model quality review.
06 — Implementation roadmap
● Phased adoption · start with lowest-trust highest-value agent

From pilot to enterprise capability

Start with Phase 1 only. It requires zero cross-boundary connectivity, no vendor contractual changes, and no MAS JIT policy modifications. It is purely an internal observability improvement that builds the evidence base — MTTR data, diagnostic quality, agent trace logs — that justifies Phase 2 investment.

01
Diagnostic Sentry (low-trust)
Months 1–3 · no cross-boundary
  • Deploy Sentry Agent inside FI boundary — read-only only, zero write access
  • Connect HPE iLO + APM (AppDynamics/Dynatrace) via MCP read tools
  • Validate PII scrubber against adversarial test suite before any cross-boundary test
  • Deploy LangSmith observability — trace every agent call from Day 1
  • Deploy OPA policy engine with deny-by-default from Day 1
  • Establish MTTR baseline measurement before any automation
  • Write ADR-001 through ADR-003 — document before building
  • Zero automated write actions in this phase — observe only
02
Specialist + Coordinator mesh
Months 4–8 · cross-boundary enabled
  • Deploy Specialist Agent in vendor environment — requires vendor contractual update for agent access
  • Establish MCP channel for Diagnostic Package transfer — validate data residency compliance with MAS counsel
  • Deploy Coordinator Agent — Teams + ServiceNow + email fan-out with shared incident_id
  • Implement JIT pre-staging via PAM API — measure dead zone reduction
  • HITL gate on all write actions — no autonomous writes for first 90 days
  • First full cross-party incident drill — run against a synthetic incident, not live production
  • Measure MTTR delta against Phase 1 baseline
  • Deploy Evaluator Agent — 72h post-fix metric monitoring
03
Full autonomous mesh
Months 9–12 · selective autonomy
  • Promote known-safe change categories to Act-with-reduced-oversight (still logged, analyst can review async)
  • Integrate Investigator Agent with SI ITSM — requires SI contractual update
  • Deploy on-prem LLM if MAS counsel determines hosted API is non-compliant
  • Build automated quarterly MAS audit report from audit ledger
  • Extend to fraud (FRAML) workflows — same architecture, new typology domain
  • Evaluate A2A protocol for direct agent-to-agent communication (Google/IBM GA expected Q3 2026)
  • Complete arc42 documentation — required for compliance sign-off on full autonomy
  • Establish quarterly Architecture Review (AAR) — review ADRs against observed production behaviour
The decision to make this week

Identify one workflow that is: (a) repetitive, (b) multi-step, and (c) currently bottlenecked by manual coordination between parties. That is your Phase 1 pilot. Do not begin with any agent that has write access. The Diagnostic Sentry is read-only by design — this is deliberate. Build the evidence base first. A single Sentry Agent that correctly diagnoses 10 incidents before humans review them is worth more than a full mesh that analysts don't trust. Trust is earned step by step, incident by incident.
